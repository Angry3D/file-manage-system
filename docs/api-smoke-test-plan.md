# 接口级 Smoke Test 评估方案（Todo 7）

最后更新：2026-05-31  
语系：cn

本文档只做评估与计划，不新增无法运行的测试脚本，不直接落地自动化。

## 1. 评估目标

1. 明确接口级 smoke test 的最小前置条件。
2. 给出可执行的覆盖范围与分阶段落地路线。
3. 标注当前阻塞项与暂停条件，避免产出“看似存在但无法运行”的测试代码。

## 2. 当前现状

1. 仓库已有构建/语法类验证（`verify:*`），但不覆盖真实数据库与上传链路。
2. 服务端当前有 `ava` 测试基础（`server/package.json` 的 `npm test`），但仓库未附带 SQL schema 和初始账号。
3. 图片主链路依赖以下外部条件：
   - MySQL 可用且有完整表结构与测试数据；
   - 上传目录可写；
   - 文件服务可访问；
   - 管理端 token 流程可用。

结论：
1. 当前可以先完成“测试计划与前提清单”。
2. 当前不应直接提交自动化脚本，否则高概率无法在他人环境运行。

## 3. 最小前置条件清单

### 3.1 基础环境

1. Node.js / pnpm 版本符合仓库要求。
2. 可执行 `pnpm install` 并通过依赖安装。

### 3.2 数据与账号

1. 可用测试数据库实例（建议独立于开发/生产）。
2. `admin_image`、`admin_upload_image` 及关联表完整 schema。
3. 至少一个可用于管理端接口鉴权的测试账号（或可稳定获取 token 的机制）。

### 3.3 文件与上传

1. `BABYLIFE_UPLOAD_DIR` 指向测试可写目录。
2. 文件服务可访问（与 `BABYLIFE_FILE_HOST` 对应）。
3. 若测试涉及真实图片处理，Sharp 相关依赖可正常工作。

### 3.4 配置与隔离

1. 明确测试环境配置注入方式（环境变量或独立配置文件）。
2. 明确测试数据隔离策略（命名空间、固定前缀、可回收数据）。

## 4. 推荐覆盖范围（Smoke 级）

以下仅列最小闭环，不追求全量分支覆盖：

1. 鉴权基础：
   - 未登录访问管理端接口应失败。
   - 有效 token 可访问受保护接口。
2. 图片新增链路：
   - 上传接口返回可用 `url/url_thumb/id`（至少基本断言）。
   - 新增图片接口成功写入，并能在列表查询到。
3. 图片编辑链路：
   - 编辑接口成功更新 `image/image_thumb/image_id/status/place/note`。
4. 图片删除链路：
   - 删除接口返回成功，列表不可再查到对应业务记录。
5. H5 展示链路：
   - `front/image/getList` 仅返回展示状态数据，且分页结构正确。

## 5. 工具路线评估

### 方案 A：沿用 `ava` + HTTP 请求库（推荐起步）

优点：
1. 与现有服务端测试栈一致，学习和接入成本低。
2. 可先做最小 smoke 用例，逐步扩展。

缺点：
1. 需要自行管理测试前后置数据准备与清理。
2. 对上传场景的文件夹隔离要求高。

### 方案 B：`supertest` 驱动应用实例

优点：
1. 便于在进程内直连服务，提高执行速度。

缺点：
1. ThinkJS 3 当前项目结构下，集成复杂度和侵入性需额外评估。
2. 文件服务与真实上传目录联动仍需外部条件。

### 方案 C：独立脚本 smoke runner

优点：
1. 逻辑灵活、易按业务流程编排。

缺点：
1. 容易与现有测试体系割裂，维护成本高。
2. 若无统一约束，最容易出现“能跑但不稳定”的脚本。

结论：
1. 优先方案 A。
2. 方案 B/C 仅在 A 无法满足时再评估。

## 6. 分阶段落地建议

### Phase 0：前提补齐（必须先做）

1. 补充 schema 和测试账号获取方式（文档化）。
2. 确认测试环境变量模板与注入方式。
3. 定义测试数据命名与回收规则。

### Phase 1：最小可运行 smoke（3~5 条）

1. 鉴权、图片新增、图片编辑、图片删除、H5 列表最小闭环。
2. 用例通过标准：稳定通过且可重复执行。

### Phase 2：稳定性增强

1. 增加失败场景断言（参数非法、token 过期等）。
2. 增加上传场景异常分支（格式、处理失败）验证。

## 7. 阻塞项与暂停条件

阻塞项：
1. 缺少 schema。
2. 缺少初始账号或 token 获取机制。
3. 缺少独立测试数据库与可写上传目录。

暂停条件（任一命中即暂停脚本落地）：
1. 无法保证测试数据隔离，存在污染风险。
2. 无法保证上传目录在测试后可控回收。
3. 用户未确认可接受的测试环境成本与维护方式。

## 8. 本轮边界说明

1. 本轮仅产出评估文档。
2. 未新增任何自动化测试脚本。
3. 未执行数据库清理、物理文件删除或高风险操作。
