# 图片资产生命周期策略评估（Todo 6）

最后更新：2026-05-31  
语系：cn

本文档只做策略评估，不包含任何业务代码变更，不执行物理文件删除，也不清理数据库记录。

## 1. 评估目标

1. 梳理当前图片资产在业务记录、上传记录、物理文件三个层面的关系。
2. 识别删除图片与替换图片场景下的数据一致性和误删风险。
3. 给出可选生命周期策略，并明确推荐方案与暂停条件。

## 2. 当前结构与关系（基于已确认代码事实）

1. 业务图片记录：`admin_image` 语义模型（ThinkJS `dbAdmin("image")`）。
   - 关键字段：`image`、`image_thumb`、`image_id`、`status`、`place`、`note`。
2. 上传记录：`admin_upload_image` 语义模型。
   - 关键字段：原文件名、相对路径、缩略图相对路径。
3. 物理文件：
   - 存储在 `Define.fileDir`（默认开发 `/Users/relax/Documents/upload`）。
   - 文件访问通过 `Define.fileHost + 相对路径`。

关系推断：
1. `admin_image.image_id` 指向上传记录语义 ID，用于关联业务图片与上传资产。
2. 同一上传资产理论上可能被多个业务记录引用（仓库内无唯一性约束证据）。
3. 当前删除图片仅删业务记录，不会自动删上传记录或物理文件。

## 3. 生命周期关键场景

### 场景 A：删除业务图片记录

现状：
1. 仅删除 `admin_image` 记录。
2. 上传记录与物理文件保留。

风险：
1. 长期累积孤儿上传记录和孤儿文件，占用存储。
2. 但误删风险低，不会影响其他可能引用同一图片的记录。

### 场景 B：编辑时替换图片

现状：
1. `setAction` 会更新 `image`、`image_thumb`、`image_id`。
2. 旧上传记录和旧物理文件保留。

风险：
1. 会产生历史资产沉淀，难以区分“历史保留”与“无人引用”。
2. 若直接删除旧文件，存在仍被其他记录引用的误删风险。

### 场景 C：上传后未入库（中断流程）

现状：
1. 上传接口成功后已写上传记录和物理文件。
2. 若用户未完成新增/编辑提交，会产生未被 `admin_image` 引用的孤儿资产。

风险：
1. 存储增长和管理复杂度上升。
2. 需要额外策略区分“短期待使用”与“长期孤儿”。

## 4. 策略选项对比

### 方案 1：保守保留（不自动清理）

定义：
1. 删除/替换都只改业务记录。
2. 上传记录与物理文件长期保留。

优点：
1. 实现简单、低风险、几乎无误删可能。
2. 便于问题追溯和人工恢复。

缺点：
1. 存储成本持续上升。
2. 需要后续人工巡检或离线治理。

适用条件：
1. 现阶段缺少 schema、引用约束和审计机制。
2. 优先保证数据安全而非空间效率。

### 方案 2：标记清理（推荐）

定义：
1. 删除/替换时不直接删除物理文件。
2. 为候选孤儿资产打标记（例如“待清理”“最后引用时间”）。
3. 通过人工审核或受控任务分批清理。

优点：
1. 相比自动删除更安全，可审计、可回滚。
2. 可逐步积累真实数据，验证清理规则有效性。

缺点：
1. 需要补充标记字段或外部台账（当前仓库无 schema，需先决策）。
2. 运维流程复杂度高于方案 1。

适用条件：
1. 用户愿意接受“先标记后清理”的治理节奏。
2. 能提供最小审计能力（清理清单、审批记录、回滚窗口）。

### 方案 3：自动删除（当前不建议落地）

定义：
1. 删除业务记录或替换图片后，立即删除旧上传记录和物理文件。

优点：
1. 存储最节省，数据面“看起来”最整洁。

缺点：
1. 误删风险最高，且通常不可逆。
2. 当前缺少引用计数、唯一性约束、回收站和回滚机制。

适用条件：
1. 仅在引用关系可严格证明、且具备完备备份/回滚后考虑。

## 5. 推荐策略

推荐：
1. 短期采用方案 1（保守保留），避免误删。
2. 中期推进方案 2（标记清理）作为过渡。
3. 暂不采用方案 3（自动删除）。

理由：
1. 当前仓库缺少 SQL schema 与约束定义，无法证明自动删除安全。
2. TODO-0005 当前阶段目标是质量提升，不是高风险数据治理落地。
3. 用户资产安全优先级高于存储优化。

## 6. 进入实现前的必备前提

1. 提供数据库 schema（至少 `admin_image`、`admin_upload_image` 结构与索引）。
2. 明确引用关系规则（是否允许多业务记录复用同一上传资产）。
3. 明确清理 SLA（延迟天数、审核角色、回滚窗口）。
4. 提供备份与恢复方案（数据库 + 文件目录）。
5. 在测试环境完成全链路演练并通过用户确认。

## 7. 暂停条件（任一命中则暂停实现）

1. 无法确认引用关系是否唯一。
2. 无法提供回滚机制或最近备份。
3. 缺少可验证的测试环境与样本数据。
4. 用户未明确授权执行清理策略。

## 8. 后续建议（仅建议，不在本轮执行）

1. 先补充 schema 文档和数据字典，再讨论字段级标记方案。
2. 先做“只读巡检报表”（识别疑似孤儿），不做删除动作。
3. 等用户确认后，再进入“标记清理”最小实现设计。
